<p>In November we released the<a href="https://medium.com/@fastifyjs/fastify-v2-is-coming-to-town-ce279738bed6"> first Release Candidate of Fastify v2</a>, and we never thought it would take us more than 3 months to be able to graduate v2 out of the RC phase. 6 releases and 85 pull requests later, here we are!</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1000/0*TFNv_SWuZQJSAzJB" /><figcaption>Courtesy of <a href="https://unsplash.com/@miklevasilyev">Mikhail Vasilyev</a></figcaption></figure><p>In these three months, we focused on porting all modules maintained in the <a href="https://github.com/fastify">Fastify github organization</a> to v2, and we were able to make more changes and bug fixes that would be breaking changes.</p><p>You can install it now with npm install fastify. Most plugins would need to be updated, but there will be a new update for them as well!</p><p>Speaking of new features and breaking changes, following you can find the list of all the major changes we did, with the link to their pull request, a brief description, and a code example.</p><p>Let us know about any compatibility and upgrade issues at <a href="https://github.com/fastify/help.">https://github.com/fastify/help.</a></p><h3>Added onError hook <a href="https://github.com/fastify/fastify/pull/1236">#1236</a></h3><p>Add a new application hook called onError, which a developer can use to track all the errors that Fastify is sending to the user. This can be super useful also for sending errors to an external tracer or to support an APM <em>(Application Performance Monitoring)</em> solution.<br />In Fastify v1 we can’t do that easily because the setErrorHandler can be overwritten and the onSend hook receives the already serialized payload.<br />In Fastify v1 there is no way to get the error object and be 100% sure that your code will never be overwritten.</p><pre>fastify.addHook(‘onError’, async (request, reply, error) =&gt; {<br /> // some code<br />})</pre><h3>Move all _ prefixed properties to Symbols in next <a href="https://github.com/fastify/fastify/pull/1245">#1237</a> <a href="https://github.com/fastify/fastify/pull/1128">#1128</a></h3><p>Internal fastify properties that were prefixed with _ in key objects, such as Request and Reply, are now using Symbols. This provides a more clear separation between the internals and your application, clearly identifying which properties you can use.</p><h3>Plugins can now receive both /prefix and prefix/ <a href="https://github.com/fastify/fastify/pull/1228">#1228</a></h3><p>The ability to prefix all the routes defined in a plugin is a key feature of Fastify. In v2 we have resolved a problem that made it impossible to set up a route for both /prefix and /prefix/, as follows:</p><pre>const fastify = require(‘fastify’)()</pre><pre>fastify.register(async function (instance, options) {<br />  // the route will be called for both ‘/english/’ and ‘/english’<br />  instance.get(‘/’, async (req, reply) =&gt; {<br />    return { greet: ‘hello’ }<br />  })<br />}, { prefix: ‘/english’ })</pre><pre>fastify.listen(8000)</pre><h3>Default error handler that logs the error <a href="https://github.com/fastify/fastify/pull/1135">#1135</a></h3><p>In Fastify v1, we logged all errors independently if there was (or not) a custom error handler (via setErrorHandler()). In Fastify v2, this behavior has been moved to the error handler. This means that an error handler is now responsible for logging the error itself.</p><pre>fastify.setErrorHandler(function (error, request, reply) {<br />  request.log.warn(error) // this is needed now<br />  var statusCode = error.statusCode &gt;= 400 ? error.statusCode : 500<br />  reply<br />    .code(statusCode)<br />    .type(‘text/plain’)<br />    .send(statusCode &gt;= 500<br />            ? ‘Internal server error’<br />            : error.message<br />    )<br />})</pre><h3>Ensure Reply#serialize uses custom serializer <a href="https://github.com/fastify/fastify/pull/1163">#1163</a></h3><p>reply.serialize(payload) serializes the payload using the custom serializer, if one is set.</p><h3>Better error management <a href="https://github.com/fastify/fastify/pull/1168">#1168</a></h3><p>We can sum up this new feature with one word: <em>standardization</em>!<br />From Fastify v2 the errors will no longer be generated inline, but they will be instances of custom errors classes.<br />You can access the full error list from require('fastify/errors') .</p><h3>Bumped avvio to 6.0.0 <a href="https://github.com/fastify/fastify/pull/1201">#1201</a></h3><p>The new version of <a href="https://github.com/mcollina/avvio">avvio</a> removes the instance from the onClose hook as the first parameter.</p><pre>fastify.addHook('onClose', (done) =&gt; {<br />  // some code<br />  done()<br />})</pre><h3>Added preParsing hook <a href="https://github.com/fastify/fastify/pull/1337">#1377</a></h3><p>This hook was needed because there was no way to run the authentication logic before the validation but after the onRequest hook or the middlewares. There are cases where you may not want to expose your validation logic to unauthenticated requests, this hook fixes that issue.</p><pre>fastify.addHook(‘preParsing’, async function (request, reply) {<br />  // request and reply are the core Fastify objects<br />})</pre><h3>Added preValidation hook <a href="https://github.com/fastify/fastify/pull/1153">#1153</a></h3><p>This hook was needed because there was no way to run execute logic after parsing a payload but before the validation. There are cases where you may want to massage your data before validation, as an example do some security checks.</p><pre>fastify.addHook('preValidation', async (request, reply) =&gt; {<br />  // some code<br />})</pre><h3>Added preSerialization hook and route option <a href="https://github.com/fastify/fastify/pull/1408">#1408</a></h3><p>If you are using the `preSerialization` hook, you can change (or replace) the payload before it is serialized. For example:</p><pre>fastify.addHook('preSerialization', <br />  async (request, reply, payload) =&gt; {<br />    return { wrapped: payload }<br />  })</pre><h3>Change basePath to prefix <a href="https://github.com/fastify/fastify/pull/1183">#1183</a></h3><p>A basePath property was populated with the prefix option when passed to a plugin. This has now been renamed to prefix, to provide a more intuitive API. The use of basePath is now deprecated.</p><h3>Separate logging and reqId options <a href="https://github.com/fastify/fastify/pull/1164">#1164</a> <a href="https://github.com/fastify/fastify/pull/1166">#1166</a></h3><p>From Fastify v2 you must create the custom request id as a top-level option in the Fastify configuration, and no longer inside the logger configuration. This is still supported, but Fastify will log a warning. The option will be dropped from Fastify v3.</p><h3>Change the default plugin timeout to 10 seconds instead of 0 <a href="https://github.com/fastify/fastify/pull/1145">#1145</a></h3><p>Each plugin now has 10 seconds to load or it will error. This solves a long-standing issue with Fastify’s bootstrapping phase: forgetting to call a callback or resolving a promise would lead to a stuck load phase with no way to easily debug it.</p><h3>Pino v5 <a href="https://github.com/fastify/fastify/pull/1065">#1065</a></h3><p>Fastify now uses Pino v5 internally, for better and more performant logging with zero cost log redaction. Check it out at <a href="https://www.nearform.com/blog/announcing-pino-v5-0-0/"><em>https://www.nearform.com/blog/announcing-pino-v5-0-0/</em></a>.</p><p>As an example, the following snippet will print all the incoming headers but the authorization.</p><pre>const fastify = require(‘fastify’)({<br />  logger: {<br />    stream: stream,<br />    redact: [‘req.headers.authorization’],<br />    level: ‘info’,<br />    serializers: {<br />      req(req) {<br />        return {<br />          method: req.method,<br />          url: req.url,<br />          headers: req.headers,<br />          hostname: req.hostname,<br />          remoteAddress: req.ip,<br />          remotePort: req.connection.remotePort<br />        }<br />      }<br />    }<br />  }<br />})</pre><h3>Drop @types/pino from dependencies <a href="https://github.com/fastify/fastify/pull/1366">#1366</a></h3><p>Fastify now ships its own types for the logger because there were discrepancies in the implementation when the logger was enabled or not. If access to the full pino API is needed, it is still possible to cast them using <a href="https://www.npmjs.com/package/@types/pino">@types/pino</a>. As a good bonus, removing this dependency shrinked the size of our bundle.</p><h3>Updated Not found handling <a href="https://github.com/fastify/fastify/pull/1084">#1084</a></h3><p>If one of your routes returns a 404 error, from Fastify v2 will be handled by the customErrorHandler and not the notFoundHandler. You can still route requests to the notFoundHandler via reply.callNotFound(). The notFoundHandler will be used only for router generated 404s.</p><h3>Hooks: always use Fastify core objects <a href="https://github.com/fastify/fastify/pull/1066">#1066</a></h3><p>This was a long-awaited “feature”, in Fastify v1 onRequest and onResponsehave different signature and objects, which leads to confusion and slightly complex internals.</p><p>From Fastify v2 we have fixed that, and now every hook will expose the exact same signature. The middlewares will continue to expose the default Node.js core objects to allow compatibility with Express middlewares. If you are not using Express middlewares, we highly recommend using Fastify hooks.</p><pre>fastify.addHook(‘onRequest’, async function (request, reply) {<br />  // request and reply are the core Fastify objects<br />})</pre><pre>fastify.addHook(‘onResponse’, async function (request, reply) {<br />  // request and reply are the core Fastify objects<br />})</pre><pre>fastify.use(function (req, res, next) {<br />  // req and res are the Node core objects<br />  next()<br />})</pre><h3>Remove `req.id` in serializer <a href="https://github.com/fastify/fastify/pull/1032">#1032</a></h3><p>There is no more the req.id property in the logs, but only reqId <em>(less duplication, less space, more speed!)</em>.</p><h3>Removed url.parse and added custom querystring parser support <a href="https://github.com/fastify/fastify/pull/1270">#1270</a></h3><p>Since url.parse will be deprecated from Node.js v12, we decided to remove it from the codebase and use the Node.js’s core querystring module instead. With this change, the querystring parsing is twice faster than before!<br />We have also exposed a new configuration option (querystringParser) to change the default querystring parser if needed.</p><h3>Extended support of cross-platform builds and yarn official support <a href="https://github.com/fastify/fastify/pull/1226">#1226</a></h3><p>Due to the extended support of <a href="https://github.com/marketplace/azure-pipelines">Azure pipelines</a>, our CI is more performant and we are able to test cross-platform <em>(Linux/yarn and Windows/npm)</em> with a single setup in a timely fashion. This also brings official yarn support for Node 8 and Node 10. We thank <a href="https://www.appveyor.com/">AppVeyor</a> for the support in the last year of Windows testing. We are also keeping our <a href="https://github.com/marketplace/travis-ci">travis-ci</a> integration for Linux/npm for the time being.</p><h3>Add plain text parser to core <a href="https://github.com/fastify/fastify/pull/1280">#1280</a></h3><p>Fastify now support parsing the ‘text/plain’ content-type out of the box.</p><h3>Default ajv instance to include all errors <a href="https://github.com/fastify/fastify/pull/1398">#1398</a></h3><p>Validation errors returned from Fastify default Ajv instance will now return the full errors.</p><figure><img alt=""
