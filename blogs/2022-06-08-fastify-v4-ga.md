<figure><img alt="" src="https://cdn-images-1.medium.com/max/1000/0*-VnsT4owkp-WgG7O" /><figcaption>Photo courtesy of Ahmed Galal @ Unsplash</figcaption></figure><p>After two years we are excited to announce the GA release of the fourth major version of Fastify! This release has been more than a year in the making and we can't wait to share with you all the new features that we think will shape the developer experience in the future.</p><p>Before doing so, we would like to acknowledge the tremendous job done by the wide Fastify team and the community, Fastify v3 had been a wildly successful release (we reached more than 1 million downloads per month and counting). We provided our most extended major release ever, with more than 20 minor releases, without breaking changes, without impacting the performances and the developer experience. This is a testament to the framework's stability and the effort the team and community put into maintaining this project.</p><p>In this new major, we focused on stabilizing the API, improving the developer experience, and upgrading our core dependencies, but this is not all! Continue reading to learn more.</p><h3>Avoid the &quot;resolved with undefined&quot; problem once and for all</h3><p>Since its inception, Fastify has combined a callback-based API (`reply.send({ hello: 'world'})`) with a promise-based one (`return { hello: 'world'}`). However this has caused a lot of confusion when mixing them, resulting in `FST_ERR_PROMISE_NOT_FULFILLED` being logged.. and most people did not know how to fix it. Since v4, we are making it mandatory to `return reply`. if you want to call `reply.send` after the route handler has completed its execution.</p><a href="https://medium.com/media/744c1ea33692b09da9d310f97132c9c3/href">https://medium.com/media/744c1ea33692b09da9d310f97132c9c3/href</a><h3>New error handling composition</h3><p>This is one of the most significant improvements of this release. Before Fastify v4, the error handling was inconsistent, each custom error handler was on its own, and if it threw an error, the final response would have been different based on where in your code that error had been thrown. In Fastify v4, custom error handlers are encapsulated, and if a custom error handler throws inside a plugin, the parent one will catch that error and handle it. As a result, the errors that your application will send back will always be consistent.</p><a href="https://medium.com/media/c08db50b0ea4671e2face79dd24feb8f/href">https://medium.com/media/c08db50b0ea4671e2face79dd24feb8f/href</a><h3>New Pino Transports</h3><p>Our out-of-the-box logger Pino, has been upgraded to v8, which means that from now on Pino transports will no longer need to live in a separate process, but can take advantage of worker threads! This will vastly improve the developer experience and simplify configuration. Take a look at this <a href="https://www.nearform.com/blog/pino7-0-0-pino-transport-worker_thread-transport/">blog post</a> to know more.</p><a href="https://medium.com/media/6f1a0bd180401182096295f243331414/href">https://medium.com/media/6f1a0bd180401182096295f243331414/href</a><h3>New Pino Types</h3><p>Pino@8 ships with its own types! So you can now specify all the options for Pino straight from the Fastify factory, without installing any additional types.</p><h3>Type providers out of the box</h3><p>Type Providers are a TypeScript only feature that enables Fastify to statically infer type information directly from inline JSON Schema. They are an alternative to specifying generic arguments on routes; and can greatly reduce the need to keep associated types for each schema defined in your project.</p><a href="https://medium.com/media/fc985eca873a02b93ce9c12573d7e125/href">https://medium.com/media/fc985eca873a02b93ce9c12573d7e125/href</a><p>You can see every other change that we are shipping in Fastify v4 <a href="https://github.com/fastify/fastify/milestone/4">here</a>.</p><p>Please let us know about any compatibility and upgrade issues at <a href="https://github.com/fastify/help">fastify/help</a>!</p><h3>Faster cold start</h3><p>Fastify uses <a href="https://github.com/ajv-validator/ajv">AJV</a> under the hood for request validation and response serialization. The current bottleneck was the use of `uri-js` which was slowing down the cold start by its processing speed and its size. Fastify team developed a lighter alternative: <a href="https://github.com/fastify/fast-uri">https://github.com/fastify/fast-uri</a></p><p>By then the startup time had improved by 150%.</p><p>Reference:<br /><a href="https://github.com/ajv-validator/ajv/pull/1862">https://github.com/ajv-validator/ajv/pull/1862</a></p><h3>Route registration is now synchronous</h3><p>Due to the changes in <a href="https://github.com/fastify/fastify/pull/2954">https://github.com/fastify/fastify/pull/2954</a>, the following code would work anymore in Fastify v4:</p><a href="https://medium.com/media/e121f999febe4931c472fc71ed521912/href">https://medium.com/media/e121f999febe4931c472fc71ed521912/href</a><p>Instead, we should write</p><a href="https://medium.com/media/9252c3db142b85ec9a0a593a01fa6b85/href">https://medium.com/media/9252c3db142b85ec9a0a593a01fa6b85/href</a><p>This change enables us to provide better error reporting for route definition.</p><h3>find-my-way v5 and v6</h3><p><a href="https://github.com/delvedor/find-my-way">find-my-way</a> is Fastify router. The list of changes in find-my-way is staggering. Here are the main highlights:</p><ol><li>Fixed parameter decoding <a href="https://github.com/delvedor/find-my-way/pull/282">https://github.com/delvedor/find-my-way/pull/282</a> and <a href="https://github.com/delvedor/find-my-way/pull/207">https://github.com/delvedor/find-my-way/pull/207</a></li><li>Improved performance when using constraints.</li><li>Many optimizations, leading an increase in the number of operations per second between 89% to 298% for most cases.</li></ol><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/0*uOZojKmlANcuPgTx" /><figcaption>Image at <a href="https://user-images.githubusercontent.com/31734731/169591618-7f242c69-1a30-4bec-9f1f-a48f166e6d57.png">https://user-images.githubusercontent.com/31734731/169591618-7f242c69-1a30-4bec-9f1f-a48f166e6d57.png</a></figcaption></figure><p>All those improvements have been led by <a href="https://github.com/ivan-tymoshenko">Ivan Tymoshenko</a>, from Kyiv, Ukraine.</p><h3>Deprecation of variadic .listen signature</h3><p>The improvements to developer experience are not limited to users of Fastify. With this release, we are starting to make the Fastify contributor experience better by simplifying code where possible. This effort begins with deprecating the <a href="https://en.wikipedia.org/wiki/Variadic_function">variadic signature</a> of the fastify.listen() method. Prior to this release, all of the following invocations of this method were valid:</p><ul><li>`fastify.listen(8000)`</li><li>`fastify.listen(8000, ‘127.0.0.1’)`</li><li>`fastify.listen(8000, ‘127.0.0.1’, 511)`</li><li>`fastify.listen(8000, (err) =&gt; { if (err) throw err })`</li><li>`fastify.listen({ port: 8000, host: ‘127.0.0.1’ }, (err) =&gt; { if (err) throw err })`</li><li>And many other permutations</li></ul><p>Starting with Fastify v4, the .listen method accepts an options object and an optional callback. Any other combination of arguments will result in a deprecation warning in v4. This will ultimately simplify the code that needs to be maintained within Fastify, making contributions like listening on both IPv4 and IPv6 at the same time easier to create. It should also result in easier to maintain TypeScript types, and, as a result, better Intellisense for the .listen method.</p><p>So, with Fastify v4, the following invocations are valid:</p><ul><li>`fastify.listen()`</li><li>`fastify.listen({ port: 8000 })`</li><li>`fastify.listen({ port: 8000 }, (err) =&gt; { if (err) throw err })`</li></ul><h3>Acknowledgments</h3><ul><li>More than 3200 commits</li><li>More than 215 releases,</li><li>More than 1700 forks</li><li>More than 2 million downloads per month</li><li>213 officially recognized plugins.</li></ul><p>This wouldn't have been possible without the help of our fantastic team, which we would like to thank for all their hard work:</p><ul><li><a href="https://github.com/allevo">Tommaso Allevi</a></li><li><a href="https://github.com/Ethan-Arrowood/">Ethan Arrowood</a></li><li><a href="https://github.com/airhorns/">Harry Brundage</a></li><li><a href="https://github.com/davidmarkclements">David Mark Clements</a></li><li><a href="https://github.com/mcollina">Matteo Collina</a></li><li><a href="https://github.com/delvedor">Tomas Della Vedova</a></li><li><a href="https://github.com/StarpTech">Dustin Deus</a></li><li><a href="https://github.com/AyoubElk">Ayoub El Khattabi</a></li><li><a href="https://github.com/SerayaEryn">Denis Fäcke</a></li><li><a href="https://github.com/rafaelgss">Rafael Gonzaga</a></li><li><a href="https://github.com/zekth">Vincent Le Goff</a></li><li><a href="https://github.com/lmammino">Luciano Mammino</a></li><li><a href="https://github.com/luisorbaiceta">Luis Orbaiceta</a></li><li><a href="https://github.com/fox1t">Maksim Sinik</a></li><li><a href="https://github.com/eomm">Manuel Spigolon</a></li><li><a href="https://github.com/jsumners">James Sumners</a></li></ul><p>We also want to thank our amazing contributors, for all the bug reports, pull requests, and faith in the Fastify vision!</p><h3>Long Term Support</h3><p>Fastify v3 will reach End of Life on 30 June 2023, but until then we will continue to maintain this version with security patches and critical fixes.</p><p>Fastify v4 is covered by our LTS support cycle.
